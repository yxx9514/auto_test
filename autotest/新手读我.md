## 这套自动化测试框架怎么读（把你当成初学者）

你只需要按顺序看 6 个文件，就能理解 80% 的框架。

### 0. 你先要知道：我们想解决什么问题

自动化测试常见有 3 种写法：

- **普通 pytest 用例**：直接写 Python 断言。优点简单；缺点是复用/表达业务步骤不够“像需求”。
- **数据驱动**：同一条用例换不同数据跑。优点覆盖更多；缺点是“步骤”还是写在代码里。
- **关键字驱动**：把“步骤”写到 YAML/Excel（人类可读），代码只提供关键字动作库。优点业务人员也能看懂；缺点是要有一套关键字引擎。

UI 自动化通常会用 **PO（Page Object）模式**：
把页面操作封装到 Page 类里，让测试用例只关心业务行为。

---

### 1. 入口：pytest 从哪里开始？

先看 `tests/conftest.py`。

pytest 运行时会自动加载 `conftest.py`，这里我们做了：

- 加载配置（`settings` fixture）
- 创建 HTTP 客户端（`http_client` fixture）
- 启动 `responses` mock（`responses_mock` fixture，保证 API 用例离线可跑）
- 创建关键字上下文（`keyword_ctx` fixture，把配置 + http + mock + 关键字表组合起来）

你以后最常改这里的两个点：

- **换环境**：`configs/config.yaml`（或者设置环境变量 `AUTOTEST_CONFIG` 指向另一个 yaml）
- **加新公共 fixture**：比如数据库连接、登录 token、测试账号等

---

### 2. 配置：我该改哪个文件来切换环境？

看 `framework/core/config.py` + `configs/config.yaml`。

- `configs/config.yaml` 是给人看的
- `framework/core/config.py` 负责把 yaml 变成 `Settings` 对象（Python 里好用）

---

### 3. API：最简单的 API 用例长什么样？

看 `tests/api/test_user_api.py` + `data/user_data.yaml`。

重点：

- `data/user_data.yaml` 里写多组用户数据
- `pytest.mark.parametrize` 让用例按每组数据各跑一次
- `responses_mock.add(...)` 让请求不出网（离线也稳定）

---

### 4. 关键字驱动：YAML 步骤是怎么跑起来的？

看三件东西：

- `data/keyword_cases.yaml`：步骤长什么样（人写的）
- `framework/keywords/engine.py`：引擎怎么解析 `${变量}`、怎么按顺序执行步骤
- `framework/keywords/builtin.py`：有哪些关键字（动作库）

关键字驱动的核心就是一句话：

> YAML 里写 `keyword`，代码里提供同名函数去执行。

你要扩展一个新关键字，只需：

1) 在 `framework/keywords/builtin.py` 新写一个 `kw_xxx(ctx, args)` 函数  
2) 在 `default_registry()` 里注册 `"xxx": kw_xxx`  
3) 在 YAML 里用 `keyword: xxx`

---

### 5. UI + PO：为什么要 Page Object？

看 `pages/base_page.py` + `pages/example_home_page.py` + `tests/ui/test_example_ui.py`。

你会发现：

- 用例里只写 `home.open()`、`home.title()`（业务语言）
- 页面细节放在 `Page` 类里（选择器、点击、输入等）

这样页面结构变了，你只改 Page，不用全项目找替换。

---

### 6. 建议你下一步怎么练？

从简单到困难：

1) 先复制 `tests/api/test_user_api.py`，做一个你自己的 API（仍然用 mock）
2) 把这条 API 用例改成“关键字驱动”的方式（写到 `keyword_cases.yaml`）
3) 装 Playwright 后，把 UI 用例也跑起来
